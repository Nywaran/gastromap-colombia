Phase 3

Model Engineering

Algorithm selection, training process, hyperparameter tuning, and validation

3.1 Algorithm Selection

Unsupervised Learning — Weighted Composite Scoring with Feature Engineering

No labeled training data exists classifying Colombian municipalities by gastronomic tourism potential. Without a target variable (Y), supervised methods cannot be applied. We construct the target through feature engineering and domain-informed weighting.

Why This Approach Was Chosen

No Ground Truth Labels

There is no existing dataset that says "Bogota = High potential, Chía = Medium." Supervised classification (Random Forest, SVM, Logistic Regression) requires labeled examples to train on. These labels do not exist for this problem.

Ranking Over Grouping

Clustering (K-Means, DBSCAN) groups data by similarity but does not produce ordinal rankings. Stakeholders need to know WHICH municipalities rank highest, not just which ones are similar. A composite index directly answers this question.

Interpretability

A weighted linear combination is fully transparent — every component's contribution is visible and adjustable. Black-box models would obscure why a municipality scored high or low, reducing trust and actionability.

Domain Expert Alignment

Weights are assigned based on domain knowledge of gastronomic tourism. Gastronomy density (35%) matters more than climate comfort (15%) because it directly measures the core variable. This mirrors how tourism experts prioritize factors.

3.2 Training Process

Since this is an unsupervised scoring model, "training" consists of the feature engineering and scoring pipeline, not gradient descent or parameter fitting.

1

Data Ingestion

Load DIVIPOLA (1,122 municipalities) and RNT (686,967 records) from disk.

2

Gastronomy Filtering

Apply regex GASTRO|BAR(?!RAN) to extract 13,237 gastronomy establishments.

3

Feature Aggregation

GROUP BY municipality code → compute gastro_count, gastro_variety, gastro_employees, total_providers.

4

Climate Estimation

Classify each municipality into a natural region → assign temperature, precipitation → compute climate_index.

5

Dataset Merging

Three LEFT JOINs on muni_code: DIVIPOLA+Climate + Gastronomy + Tourism. Fill nulls with 0.

6

Log Transformation

Apply log(1+x) to gastro_count, total_providers, gastro_employees to handle extreme skew.

7

Min-Max Normalization

Scale all 5 components to [0, 1]: x_norm = (x - x_min) / (x_max - x_min).

8

Weighted Scoring

GTPI = 0.35×GD + 0.20×GV + 0.20×TI + 0.15×CC + 0.10×GE → scale to 0-100.

9

Classification

Equal-width binning into 5 tiers: Very Low, Low, Medium, High, Very High.

The entire pipeline runs in approximately 8 seconds on a standard laptop, processing 686,967 input records into 1,015 scored municipalities.

3.3 Hyperparameters

The model's "hyperparameters" are the component weights and classification bin edges:

ParameterValueJustification
GD Weight0.35Highest — directly measures core variable (gastronomic offering)
GV Weight0.20Variety enriches tourist experience — second tier importance
TI Weight0.20Infrastructure enables tourism — equal to variety
CC Weight0.15Climate is contextual, not central to gastronomy
GE Weight0.10Employment is a secondary indicator of sector maturity
Bin edges[0, 20, 40, 60, 80, 100]Equal-width for intuitive interpretation
Log baselog(1+x)Natural log with +1 shift to handle zeros
Climate optimal T23°CCenter of comfort range (18-28°C)
Climate optimal P1500 mm/yrModerate rainfall — not too dry, not too wet

In future iterations, these could be optimized using AHP (Analytic Hierarchy Process) with expert pairwise comparisons, or PCA to derive data-driven weights.
