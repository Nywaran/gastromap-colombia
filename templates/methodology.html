Methodology

Machine Learning approach, data treatment, and scoring model

1. Type of Machine Learning Used

Unsupervised Learning — Feature-Engineered Composite Scoring

This system uses an unsupervised machine learning approach because there is no labeled training data telling us which municipalities are "good" or "bad" for gastronomic tourism. Instead, we engineer features from raw data and combine them into a weighted composite index.

Why Unsupervised?

There is no existing labeled dataset that classifies Colombian municipalities by gastronomic tourism potential. Supervised approaches (classification, regression) require labeled examples to learn from. Since these labels do not exist, we must construct the target variable ourselves through feature engineering and domain-informed weighting — a classic unsupervised scoring problem.

Why Not Clustering (K-Means, DBSCAN)?

Clustering algorithms group data by similarity but do not produce an ordinal ranking. For this use case, stakeholders need to know which municipalities rank highest, not just which ones are similar. A composite index provides an interpretable, sortable score that directly answers the business question.

Specific Techniques Applied

Feature Engineering: aggregation (count, nunique, sum), log-transformation (log1p) to handle skewed distributions, and domain-based climate modeling. Normalization: min-max scaling to [0,1] for cross-feature comparability. Weighted Linear Combination: domain-expert-informed weights produce the composite GTPI score. Binning: equal-width discretization into 5 classes.

Scalability to Supervised Learning

The GTPI scores generated by this system can serve as pseudo-labels for future supervised models. Once domain experts validate and refine the top/bottom classifications, a gradient boosting or random forest classifier could be trained to predict GTPI class from raw features, enabling automated scoring of new municipalities.

2. Data Sources

DIVIPOLA — DANE (National Administrative Department of Statistics)

The Political-Administrative Division Code provides a unique 5-digit code for each of Colombia's 1,122 municipalities, along with geographic coordinates (latitude, longitude), department name, and municipality type. This serves as the primary join key for all dataset merges.

Geographic .xlsx · 1,122 records · 8 columns

National Tourism Registry (RNT) — MinCIT

Complete registry of all tourism service providers operating in Colombia. Each record contains the provider's category (accommodation, gastronomy, travel agency, transport, etc.), subcategory, municipality code, number of employees, and registration year. This is the richest dataset in the pipeline.

Tourism .csv · 686,967 records · 14 columns

Climate Estimation Model — Coordinate-Based

Colombia's unique geography means temperature is primarily determined by altitude (thermal floors: cálido, templado, frío, páramo), not latitude. We estimate climate variables using a rule-based model that classifies each municipality into one of five natural regions (Andean, Caribbean, Pacific, Orinoquia, Amazon) based on its coordinates, then assigns baseline temperature and precipitation values with latitude corrections.

Climatic Computed from DIVIPOLA coordinates

3. Data Treatment — Step by Step

3.1 — Loading & Cleaning

DIVIPOLA

The Excel file has 10 header rows (title, subtitle, blank lines) before the actual data. We skip these and manually assign column names. Department codes are zero-padded to 2 digits, municipality codes to 5 digits (e.g., 5001 → 05001). The department name column contains merged cells in the original Excel, so we apply forward-fill (ffill) to propagate department names down to every municipality row. Rows with null latitude or longitude are dropped.

RNT

The CSV is loaded with low_memory=False to prevent mixed-type column inference on the 686K-row file. Municipality and department codes are zero-padded to match DIVIPOLA. All records have ESTADO_RNT = ACTIVO, so no status filtering is needed. No null values exist in key columns (category, municipality code).

3.2 — Gastronomy Filtering

From the 686,967 RNT records, we select gastronomy-related establishments using a regex filter on the CATEGORIA column:

GASTRO|BAR(?!RAN)

The pattern BAR(?!RAN) matches "BAR" and "BARES" but uses a negative lookahead to exclude "BARRANQUILLA" (which would create false matches in municipality names). This yields 13,237 gastronomy records across 371 municipalities with 25 distinct subcategories (Restaurante, Gastrobar, Café, Cevichería, Panadería, Heladería, etc.).

3.4 — Climate Estimation

Each municipality is assigned to a natural region using coordinate-based rules:

if latitude > 8° → Caribbean elif longitude < -76.5° → Pacific elif longitude > -72° → Orinoquia elif lat < 2° & lon > -73° → Amazon else → Andean

Baseline temperature and precipitation are assigned per region, then temperature is adjusted by latitude (warmer further north: +0.3°C per degree). A climate comfort index (0 to 1) is computed as:

comfort = 0.6 × temp_comfort + 0.4 × precip_comfort

Where temp_comfort = 1 - |T - 23| / 15 (optimal at 23°C) and precip_comfort = 1 - |P - 1500| / 4000 (optimal at 1500 mm/year), both clipped to [0, 1].

4. Data Merging — When and How

Step 1 — Base: DIVIPOLA + Climate

The climate estimation runs directly on the DIVIPOLA dataframe, adding region, avg_temperature, precipitation, and climate_index columns. This produces 1,122 rows (one per municipality).

In-place computation (no join needed)
Step 2 — LEFT JOIN: Base + Gastronomy Aggregation

A left join on muni_code adds gastronomy features. Municipalities without gastronomy retain null values (later filled with 0). This preserves all 1,122 municipalities.

LEFT JOIN on muni_code (5-digit DIVIPOLA)
Step 3 — LEFT JOIN: Result + Tourism Aggregation

A second left join on muni_code adds general tourism features (total_providers, total_employees, unique_categories).

LEFT JOIN on muni_code (5-digit DIVIPOLA)
Step 4 — Null Handling & Activity Filter

All null numeric columns are filled with 0 (municipality has no registered gastronomy/tourism). Then we filter to keep only municipalities with total_providers > 0, reducing from 1,122 to 1,015 active municipalities. The 107 excluded have zero tourism activity of any kind.

fillna(0) → filter(total_providers > 0)

5. GTPI Scoring Model

The Gastronomic Tourism Potential Index (GTPI) is a weighted linear combination of five normalized features, producing a score from 0 to 100.

GTPI = 0.35 × GD + 0.20 × GV + 0.20 × TI + 0.15 × CC + 0.10 × GE

5.1 — Transformation: Log(1+x)

Three features (gastro_count, total_providers, gastro_employees) have highly right-skewed distributions — Bogotá has 3,854 gastronomy establishments while the median is 5. Applying log(1+x) compresses the range and prevents outliers from dominating the normalized score. The "+1" handles zero values safely.

5.2 — Normalization: Min-Max Scaling

Each feature is scaled to [0, 1] using:

x_norm = (x - x_min) / (x_max - x_min)

This ensures all five components contribute proportionally regardless of their original scales (counts vs. index values).

5.3 — Weighted Combination

Weights were assigned based on domain relevance to gastronomic tourism:

35%

Gastronomy Density (GD)

Log-normalized count of gastronomy establishments. Highest weight because it directly measures the core variable: gastronomic offering.

20%

Gastronomy Variety (GV)

Number of distinct subcategories. A municipality with restaurants, gastrobars, cafés, and cevicherías scores higher than one with only restaurants — variety enriches the tourist experience.

20%

Tourism Infrastructure (TI)

Log-normalized count of all tourism providers. Good gastronomy alone is not enough — tourists also need hotels, transport, and agencies.

15%

Climate Comfort (CC)

The climate comfort index (0-1). Favorable weather encourages tourism; extreme temperatures or heavy rainfall deter visitors.

10%

Gastronomy Employment (GE)

Log-normalized employee count. Indicates sector maturity and economic impact — a municipality with more gastronomy jobs has a more established food scene.

5.4 — Final Scaling & Classification

The raw weighted sum (0 to 1) is multiplied by 100 and rounded to one decimal. Then equal-width binning assigns each municipality to a tier:

7. Limitations & Future Work

Climate Model Precision

The current model estimates temperature by region and latitude, without real altitude data. Integrating SRTM elevation rasters or WorldClim bioclimatic variables would significantly improve accuracy for Andean municipalities where altitude varies dramatically.

Static Weights

The GTPI component weights (35/20/20/15/10) are domain-informed but static. A data-driven approach using PCA (Principal Component Analysis) or AHP (Analytic Hierarchy Process) could optimize weights based on variance or expert pairwise comparisons.

Demand-Side Data

The model only considers supply (establishments, infrastructure). Incorporating visitor flow data (DANE tourism surveys, Google Trends, social media sentiment) would capture actual demand and traveler preferences.

Temporal Dimension

The RNT spans 2019-2026 but is aggregated without temporal weighting. Recent registrations could be given more weight to capture momentum — a municipality gaining 50 new restaurants in 2025 has more potential than one that peaked in 2020.
